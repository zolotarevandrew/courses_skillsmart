Очевидно, что при текстовых инструкциях LLM начинает путаться. Впрочем, как и с человеком: их можно понять по-разному.

Grace — интересный пример для рефлексии: возможность генерировать код на лету внутри функций, а также импортировать только что сгенерированный код.

Но в любом случае придется ограничивать этот код инвариантами, предусловиями и постусловиями для соблюдения корректности.

Будет ли кто-то подключать такое в продакшн-системе?

Разве что для не business-critical историй, возможно, для рекомендаций или общения.


## DSL для агентов

Если двигаться далее в сторону агентов, то для общения с конкретным агентом в идеале нужно использовать свой DSL.

Но вот дилемма: надо держать множество DSL в голове, а хочется иметь что-то условно универсальное.

Поэтому в таком общем DSL нужно иметь общие конструкции, но это уже близко к надстройке над языком программирования.

Важно, что DSL должен быть формальной спецификацией, чаще всего — декларативным описанием предметной области.

Если DSL становится императивным, он превращается в язык программирования, а задача — описать, как должен выглядеть пайплайн бизнес-функции с ограничениями.

Я думаю, лучше всего подойдет что-то в виде YAML. JSON неудобно описывать из-за вложенности скобок.

Далее берем четыре уровня по Кэю и добавляем базовый:
0. Базовые общие конструкции.
1. Данные / сущности.
2. Операции над ними.
3. Политики / правила.
4. Оркестрация / процессы.

Здесь надо поискать варианты реализации, интересная задача, начну копать.

## Разбор DSL

Приходим к Tatsu (или чему-то еще) для разбора нашего DSL в AST, но здесь же еще нужно сделать семантический анализ.

То есть флоу разработки становится таким:

1. Продакт собирает требования.

2. Аналитик пишет DSL на основе заданной грамматики 0-го уровня, а не как сейчас — непонятно что :) Нужна общая грамматика:

   * четыре уровня: данные, операции, правила — предусловия, постусловия и инварианты, — оркестрация;
   * включает туда use cases сразу для генерации тестов (предусловия, постусловия и инварианты также используются для генерации.)

3. DSL передается в Tatsu:

   * получаем AST;
   * делаем семантический анализ;
   * получаем типизированный AST.

Финальный шаг — это чудо уже можно передавать в LLM для генерации .NET-кода.

То есть аналитик потенциально может сделать MR, который разработчик будет дорабатывать :)

## DDD

Далее приходим к DDD. Чтобы сохранять сложность системы, нужно:

* прорабатывать ubiquitous language, который будет состоять из нашего DSL;
* выделять bounded contexts;
* описывать правила взаимодействия между contexts — context map?

А вишенкой на торте будет реализация с использованием ФП.
