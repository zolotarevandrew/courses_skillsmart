Изучил модули:

1) Функциональные архитектуры (неработающая инфраструктура и ECS)

Уже автоматически следую ряду принципов:
- Конфигурация в проектах всегда типизирована, никаких динамических строковых значений.
  Дополнительно попробую вариант с Capabilities (`where T : IConf1, IConf2`) для более сложных конфигураций.

2) ФА и ООП
- Действительно, для проектов с редко меняющимися бизнес-требованиями ФП-подход — лучшее решение (уже испробовал).
- Абстрактные фабрики никогда не использовал, но зато теперь понимаю, как от них избавляться совсем.
- Вынесение деталей времени жизни и конфигурации в Composition Root — уже достаточно давно практикую. Момент с декоратором взял на заметку.
- На примере DI и контроллеров стараюсь делать зависимости методов простыми и инжектировать напрямую не через конструктор, а в сам метод — .NET такое позволяет.
  Также более предпочтительным вариантом будет являться «1 контроллер — 1 метод», ближе к Minimal API.

3) ФА и система типов
- Тошнит от обилия интерфейсов в .NET ООП-проектах, поэтому периодически пытаюсь создавать делегаты, но достаточно редко.
- Reader/State Monad — удобный способ протащить состояние между шагами и сделать мини-DSL. Полезно на уровне programming in small.
- Free Monad — уже более высокоуровневый DSL с отдельным интерпретатором. Пока не придумал мест, где можно использовать, но явно, когда нужна некая оркестрация процесса или сложная обработка.

Внедрил на новом проекте:
- Алгебраические типы данных на основе собственной реализации — через вложенные record.
  Каждый раз новая, без переусложнения.

```csharp
public abstract record KubernetesJobStatus
{
    private KubernetesJobStatus() { }
    
    public record Completed() : KubernetesJobStatus;
    public record Failed(string Reason) : KubernetesJobStatus;
    public record Executing() : KubernetesJobStatus;
    
    public TResult Match<TResult>(
        Func<Completed, TResult> completed,
        Func<Failed, TResult> failed,
        Func<Executing, TResult> executing)
    {
        return this switch
        {
            Completed c => completed(c),
            Failed f => failed(f),
            Executing e => executing(e),
            _ => throw new InvalidOperationException()
        };
    }
}
```

- Самописные базовые монады Option/Either.
  Исключения используются только для реально исключительных ситуаций — валидации инвариантов.
- Переход на CQRS был избыточен. Просто следую ФП-подходу, отделяя I/O и чистые функции, бизнес-логики не так много.
