Чтобы код соответствовал OCP:
Избавляемся от Match, как и описано, из него полностью следует реализация. 
Заводим следующие типы:

```c#
// стратегия будет описана явной типизированной функцией.
public record MatchStrategy( Func<BoardState, MatchGroup> FindMatches );

// группа матчей, заводим отдельный тип для более легкого изменения/расширения в будущем, а не просто IEnumerable.
public record MatchGroup( IReadOnlyList<MatchResult> Results ); 

// делаем структурой, чтобы хранилось прямо в самом массиве, в памяти.
public struct MatchResult( IReadOnlyList<MatchPosition> Positions, MatchKind Kind );

// позиция части матча
public struct MatchPosition( int Row, int Col );

public enum MatchKind 
{
    Horizontal = 1,
    Vertical = 2,
}

// типизируем сами бонусы, сможем добавить пустой бонус + уникальные операции.
public struct Bonus( int Value );

public static class MatchStrategies {
    public static class Horizontal
    {
        public static MatchStrategy Value = new MatchStrategy(
             static state => new MatchGroup( [] )       
        );
    }
}
```

Логика следующая:
- Каждая конкретная стратегия "матчинга" описывает независимую, легко изменяемую функцию (алгоритм).
 
- Сами стратегии можно будет передавать извне, либо получать динамически из конфигурации, на основе текущего игрового состояния.
Например, в зависимости от текущих бонусов юзера, можно выдавать ему расширенный набор стратегий.

- Предыдущая реализация FindMatches просто исчезает, и ЧТО важно у нас остается ApplyCascadeStep, 
вынесенная ранее функция, мы ее легко можем переиспользовать, слегка изменив сигнатуру входных данных.
Придется изменить MarkCellsForRemoval, но за счет хранения типизированных списков позиций, это легко сделать уже в функциональном стиле.

- CalculateScore исчезает, для расчета бонусов лучше также завести отдельные стратегии, которые могут считать бонусы не только на основе матчей, но и на основе игрового состояния. 
и рассчитывать их отдельным шагом на основе MatchGroup/MatchResult/MatchKind
